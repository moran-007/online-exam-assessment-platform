# 在线答题与智能测评平台 PRD 产品需求文档

版本：v1.0  
适用范围：在线答题网站 / 题库系统 / 考试系统 / Hydro 编程题测评接入 / AI 学情分析预留

---

## 一、项目背景

本项目面向少儿编程、信息科技、算法基础、Python、C++、Scratch、Arduino 等课程场景，建设一个在线答题与智能测评平台。

平台需要支持普通题型在线答题，同时为后期编程题测评预留 Hydro 接入能力，并为 AI 学情分析、薄弱点总结、针对性练习推荐预留数据结构和接口。

系统第一阶段目标是完成：

1. 题库管理。
2. 题目分类。
3. 随机组卷。
4. 在线答题。
5. 自动判分。
6. 成绩统计。
7. 错题本。
8. 数据导出。
9. 试卷导出。
10. Hydro 和 AI 能力预留。

## 五、功能需求

### 5.1 登录认证

功能说明：

1. 用户通过账号密码登录。
2. 系统区分管理员、教师、学生等角色。
3. 登录后根据角色展示不同菜单。
4. Token 过期后支持刷新。
5. 用户退出后 Token 失效。
6. 用户可选择是否保持登录；未选择时关闭浏览器即结束本地会话，选择后允许跨浏览器重启恢复登录。
7. 默认 Access Token 有效期为 15 分钟；普通会话最长 8 小时，保持登录会话最长 7 天。
8. 连续 30 分钟无有效操作后会话失效，前后端均不得继续刷新或访问受保护接口。
9. 点击、键盘、触摸、滚动及其触发的已认证业务请求计为有效操作；后台静默任务不得无限延长会话。

验收标准：

1. 正确账号密码可以登录。
2. 错误账号密码不能登录。
3. 禁用用户不能登录。
4. 不同角色登录后菜单不同。
5. 未登录不能访问受保护页面。
6. 未勾选保持登录时令牌只保存在当前浏览器会话，勾选后才持久化保存。
7. 闲置达到配置时长后自动跳转登录页，并明确提示“登录已失效”。
8. 主动退出后，原 Access Token 与 Refresh Token 均不能再次访问或刷新。
9. 有效操作可更新服务端最后活动时间，活动上报应节流，避免高频请求。

### 5.5 题库管理

#### 5.5.1 支持题型

当前题型引擎基线支持：

1. 单选题。
2. 多选题。
3. 判断题。
4. 填空题。

#### 5.5.2 题目字段

题目应包含：

1. 题干。
2. 题型。
3. 选项。
4. 正确答案。
5. 解析。
6. 默认分值。
7. 难度。
8. 课程。
9. 知识点。
10. 标签。
11. 状态。
12. 创建人。
13. 审核人。
14. 版本号。
15. 是否允许选项随机。
16. 材料题父子关系、子题顺序和子题独立分值。
17. 评分规则版本和资源快照。

#### 5.5.3 题目状态

```txt
draft             草稿
pending_review    待审核
published         已发布
disabled          已停用
archived          已归档
```

#### 5.5.4 题目版本

需求说明：

1. 题目修改后必须生成新版本。
2. 已发布考试使用题目快照。
3. 历史考试不受题库修改影响。
4. 可查看历史版本。

验收标准：

1. 老师修改题目后，不影响已经发布的考试。
2. 历史答题记录可以还原当时题目。
3. 试卷导出应使用发布时的题目快照。

### 5.7 规则组卷

功能说明：

教师可按规则自动抽题。

规则条件包括：

1. 题型。
2. 课程。
3. 知识点。
4. 标签。
5. 难度。
6. 题目数量。
7. 每题分值。
8. 是否排除已选题。

示例：

```txt
单选题：知识点为变量和循环，难度 1-2，抽 10 题，每题 2 分。
多选题：知识点为条件判断，难度 2-3，抽 5 题，每题 4 分。
填空题：知识点为输入输出，难度 1-3，抽 5 题，每题 4 分。
```

验收标准：

1. 题库数量不足时不能发布。
2. 系统提示具体哪条规则题量不足。
3. 同一试卷中不能重复抽取同一道题。
4. 规则组卷结果可以预览。
5. 组卷后可以手动微调。

### 5.8 随机组卷

功能说明：

随机组卷分两种：

1. 老师发布时随机生成一套固定试卷。
2. 学生进入考试时，每人生成不同试卷。

每人随机卷要求：

1. 学生第一次进入考试时生成个人试卷实例。
2. 刷新页面不能重新抽题。
3. 退出后再次进入仍然读取原实例。
4. 每个学生可以看到不同题目或不同顺序。
5. 历史记录必须能还原学生实际看到的试卷。

验收标准：

1. 同一学生多次进入考试，试卷不变化。
2. 不同学生可按规则生成不同试卷。
3. 题目和选项顺序保存在个人试卷实例中。
4. 判分不受选项展示顺序影响。

当前实现说明：

1. 固定试卷保持题目顺序/选项顺序随机。
2. `random` 类型试卷在学生首次进入考试时按规则抽题并写入个人 `paper_instances`，后续刷新和再次进入复用同一实例。

### 5.10 判分

#### 5.10.1 客观题自动判分

支持：

1. 单选题自动判分。
2. 多选题自动判分。
3. 判断题自动判分。
4. 填空题自动判分。
5. 所有自动判分统一输出 `ScoreResult`，包含得分、满分、正确状态、处理状态、逐项明细、警告和引擎版本。

#### 5.10.2 多选题评分模式

支持：

1. 严格模式：全对得分，否则 0 分。
2. 少选部分分：少选且无错选，按比例给分。
3. 扣分模式：选对加分、错选扣分，最低不低于 0 分。

默认使用严格模式，可按题目评分规则切换为少选部分分或扣分模式。

#### 5.10.3 填空题评分规则

支持：

1. 完全匹配。
2. 忽略大小写。
3. 忽略首尾空格。
4. 多个可接受答案。
5. 数值误差范围。
6. 安全正则匹配。

#### 5.10.4 主观题人工批改

功能包括：

1. 查看学生答案。
2. 查看参考答案。
3. 输入得分。
4. 输入评语。
5. 保存批改结果。
6. 批量发布成绩。
7. rubric 维度评分，由服务端按维度上限求和；试卷覆盖分值时按比例调整维度上限并修正尾差。
8. 评分历史不可覆盖，人工批改、自动判分、Judge 和重判均保留规则快照、明细、操作者和时间。

#### 5.10.5 试算重判

功能包括：

1. 支持按考试、题目、学生或作答范围生成试算重判任务。
2. 试算默认使用考试快照中的评分规则，也可显式选择题目最新规则或指定规则版本。
3. 试算不会改变正式成绩，确认前展示扫描数量、变化数量、错误和警告。
4. 确认时校验答案更新时间与得分指纹；数据已变化则拒绝覆盖。
5. 兼容旧的单次重判接口，内部生成并立即确认单次重判任务。
6. AI 评分建议不能直接成为正式评价，教师必须显式提交 rubric 或人工分数。

验收标准：

1. 客观题提交后自动判分。
2. 主观题进入待批改状态。
3. 主观题批改完成后汇总总分。
4. 批改人和批改时间必须记录。
5. 成绩发布前学生不能看到未发布结果。
6. 每次正式成绩变化都能追溯到对应评分历史。

### 5.11 Hydro 编程题接入

第一阶段：

1. 编程题作为正式题型。
2. 支持绑定 Hydro 题号、题目链接、平台站点、语言配置。
3. 学生可在考试或练习中提交代码。
4. 老师可手动回写或修正判题结果。

第二阶段：

1. 平台绑定 Hydro 账号。
2. 同步 Hydro 提交结果。
3. 编程题成绩汇总到考试总分。
4. Hydro 任务级看板按考试、班级、题号聚合结果。

第三阶段：

1. Hydro 插件回调。
2. 自动同步判题状态。
3. 批量同步、失败重试、结果摘要沉淀到 `hydro_results`。
4. 自动生成编程题分析。

验收标准：

1. 平台可以创建编程题。
2. 编程题可以绑定 Hydro 地址。
3. 学生可以提交 Hydro 测评代码。
4. 平台可以保存 Hydro 结果摘要。
5. Hydro 同步失败可以重试或人工修正。

### 5.16 平台引擎化演进，结合现有实现的新增需求

#### 5.16.1 产品边界与可行性结论

“题量、题型、内容和场景不限”作为架构方向，不作为无容量上限的即时承诺。平台应通过可测量的容量、延迟、兼容性和扩展成本指标逐阶段达成该目标。

结合当前实现：

1. 已有 `questions`、`question_options`、`question_answers`、`question_versions`、`question_compositions`、评分规则版本和题目/试卷/个人试卷快照，可作为通用题目模型的迁移起点，不需推倒重建。
2. `answerJson` 和 `scoringRuleJson` 继续承载题型差异；题型注册表已经收敛编辑、作答、判分和元数据输出基线，`QuestionType` 枚举和 API 字符串保持兼容。
3. 材料题/组合题已落地单层父子题模型、子题独立分值、作答、判分、错题上下文和快照；递归组合、多层材料题暂不开放。
4. 已有 AI 报告/提示词模型、错题事件、复习规则和知识点统计，可行地支持“教师审核后发布”的 AI 学情最小闭环。
5. 已有 `ExternalOjPlatform`、Hydro 账号/任务/结果数据，多 Judge Provider 适合在保留 Hydro 兼容的前提下逐步抽象。
6. 私有资源鉴权、签名访问、引用扫描和备份恢复已建立本地存储安全基线；迁移 MinIO/S3 应通过存储适配层完成，不改变 Markdown 逻辑资源标识。
7. 动态题型定义不得从数据库加载未审核的可执行 JavaScript/Vue 组件。第一阶段只允许已发布代码中的编辑器、作答器、判分器、统计器和导出器键值白名单。
8. 多租户会影响几乎全部业务表和唯一索引。在没有确定多机构产品模式前，不盲目给所有表增加可空 `tenant_id`；先完成数据归属边界 RFC 和迁移演练。
9. 评分历史和重判任务已建立兼容双写：旧字段继续服务页面和旧接口，正式评分证据写入不可覆盖的评价历史。

#### 5.16.2 题型注册中心

新增稳定的题型定义契约，每种题型至少声明：

1. `typeCode` 和定义版本。
2. 题目内容 JSON Schema、学生答案 JSON Schema 和评分规则 JSON Schema。
3. 教师编辑器、学生作答器、预览器的白名单组件键。
4. 判分、统计和导出适配器键。
5. 是否支持自动判分、人工批改、AI 辅助或外部 Judge。
6. 允许的附件类型、大小限制和安全策略。

实施约束：

1. 当前以 TypeScript 注册表和共享接口收敛现有题型，验证稳定后再增加 `question_type_definitions` 元数据表。
2. 数据库只保存 schema、组件/适配器键和版本，不保存或执行任意代码。
3. 未识别的题型版本必须可以安全降级为只读展示，不得导致整份历史试卷无法查看。
4. 现有题型和 API 在迁移期继续可用，通过 adapter 逐个迁移；当前已接入考试、练习、题库、试卷和批改核心链路。

#### 5.16.3 材料题与组合题

第一个新引擎化题型为“材料/组合容器 + 子题”，子题复用已注册的任意非材料题型；它既用于阅读材料题，也用于多问简答题组等大题结构。

需求：

1. 容器题保存材料内容、资源引用、展示配置和子题顺序；子题保存独立题型、答案、评分规则、分值和知识点。
2. 组卷按整道材料大题加入，默认不拆分大题；子题顺序和分值在材料关系表中维护。
3. 考试发布和学生首次进入时，快照必须冻结材料、子题、子题顺序、分值、答案/评分规则版本和资源版本。
4. 子题独立保存、判分和统计，同时汇总大题得分率。
5. 错题本收录子题时保留原材料快照和大题来源。
6. 单题导入应支持在父题内联添加多个小题；Excel 导入、题目详情、试卷、考试、错题本和导出必须保持材料/大题说明与子题结构。
7. 数据模型优先采用可约束的父子关系/子题表，不仅把全部子题塞入无索引 JSON。
8. 当前只支持单层组合，禁止材料题继续嵌套及循环引用。

#### 5.16.4 判分引擎与可追溯重判

统一判分接口应支持选择严格/部分分、文本/数值/正则填空、rubric 人工评分、文件人工评分、外部 Judge 和 AI 辅助，并输出统一结果：

```txt
score
maxScore
scoringStatus
scoringMethod
scoringRuleSnapshot
scoringVersion
scoredBy
scoredAt
scoreDetail
```

要求：

1. 每次正式判分保存评分规则快照、引擎版本和可读评分明细。
2. 支持按考试、题目、学生范围试算重判，试算结果确认后才替换正式成绩。
3. 重判不覆盖旧成绩证据，必须记录发起人、原因、旧分、新分、规则版本和时间。
4. AI 可以给出评分建议和 rubric 证据，但默认不直接改变正式分数；必须由教师确认或通过明确审批策略。
5. 外部 Judge 回调必须幂等，且不得使用过期题目/规则版本覆盖新结果。

#### 5.16.5 AI 学情与推荐练习最小闭环

第一版流程：

```txt
考试/练习结果 -> 知识点得分和错题特征 -> 规则基线 -> AI 生成报告 -> 教师编辑审核 -> 发布给学生
```

需求：

1. 先用确定性统计生成得分、正确率、错题类型和知识点薄弱项，AI 只负责解释、归因候选、讲评建议和练习推荐文案。
2. AI 输入必须使用去敏快照，记录模型、提示词版本和输入/输出快照。
3. 教师可编辑、驳回、重新生成和发布；未发布报告学生不可见。
4. 增加学生-课程-知识点掌握度聚合模型，保存掌握度、作答次数、总体/近期正确率、最近练习时间和推荐难度；薄弱原因需区分规则证据和 AI 推断。
5. 推荐分三阶段：未掌握错题 -> 同知识点/相近难度题 -> 基于掌握度的自适应题组。
6. 推荐必须遵守题目可见范围、考试占用、版权和学生所属课程权限。

#### 5.16.6 题库治理、检索与开放范围

1. 题目可见范围分为 `private`、`organization`、`course_students`、`login_users` 和 `public`；`marketplace` 在版权、授权和结算模型确定后再开启。
2. 增加题目来源、复制/派生关系、版权/授权说明、引用次数、质量分、纠错/举报和审核状态。
3. 题目质量分必须展示构成因素，例如作答量、错误率、区分度、争议数和人工审核，不得只给不可解释的单一分数。
4. 短期使用 PostgreSQL 索引和全文检索支持题干、标题、解析、标签和知识点；达到容量/延迟阈值后，通过索引同步适配层迁移到 Meilisearch/Typesense/OpenSearch 中经评估选定的一种。
5. 相似题检测覆盖导入、编辑和发布审核，但相似不等于重复，必须支持人工确认与误判反馈。
6. 批量改标签、迁移知识点、归档、审核和重建索引必须通过异步任务并提供影响预览。

#### 5.16.7 资源、任务与 Judge Provider 引擎

1. 资源存储提供 Local 和 S3 兼容 Provider；逻辑资源 ID、引用权限和审计与存储实现解耦。
2. 对象存储阶段增加资源版本、引用计数、内容哈希、分片/断点续传、恶意文件扫描、下载审计和跨环境 manifest 校验。
3. 统一异步任务状态为排队、执行中、成功、失败、已取消和已过期，支持进度、重试、日志、结果鉴权下载和权限快照。
4. 题库导入/导出、AI 报告、Judge 同步/重判、资源清理、相似题检测和统计预计算逐步迁移到 Redis + BullMQ；当前单进程导出队列作为兼容过渡。
5. `JudgeProvider` 统一提供提交、查询结果、同步题目、同步提交、重判和结果归一化接口；Hydro 是首个 Provider，保留现有 API 兼容。
6. Scratch/Arduino 第一阶段以项目文件鉴权上传 + rubric 人工/半自动评分落地，不在没有可验证运行环境时宣称全自动判分。

#### 5.16.8 考试异常、细粒度权限和多租户预研

1. 轻量考试安全优先记录切屏、离开页面、复制粘贴、网络中断/恢复、异常保存/提交、IP、设备和多端会话冲突；行为记录只作为教师核查证据，不自动定性作弊。
2. 答案保存增加客户端修订号/幂等键和服务端版本，冲突时保存双方快照并向学生给出可理解提示。
3. 权限分为菜单、接口、数据范围和字段/操作四层；前端按钮隐藏不代替后端权限校验。
4. 成绩、正确答案、解析、学生身份信息、导出和资源下载分别定义字段/操作权限，并在序列化层进行服务端裁剪。
5. 多租户在 P2 开始前完成组织、校区、成员关系、公共/机构题库、唯一索引作用域、导出隔离和存量数据迁移 RFC；未通过恢复演练不执行全表租户化。

当前已落地的细分权限包括 `grading:score:read/update`、`grading:rubric:update`、`grading:regrade:preview/confirm`、`exam:answer:read`、`question:answer:read`、`question:analysis:read`、`student:identity:read`、`export:task:create/read`、`export:file:download`、`attachment:preview/download`。旧 `grading:*`、`exam:result:*`、`question:read` 仍作为兼容总权限；权限名称和内置角色名称面向用户显示中文，稳定 code 保持英文。

#### 5.16.9 分阶段优先级

P0，优先收敛平台底座：

1. 题型契约、注册表与现有题型 adapter。
2. 材料/组合题父子结构与完整快照。
3. 判分器接口、评分规则版本、评分明细和试算重判。
4. 存储 Provider 接口与 BullMQ 任务基础设施。
5. 成绩/答案/解析/导出的字段与操作权限。

P1，补齐教学闭环：

1. AI 学情报告教师审核发布。
2. 学生知识点掌握度和三层推荐练习。
3. 主观题 rubric、班级高频错题计划和教师讲评建议。
4. 多 Judge Provider 和 Judge 任务队列化。

P2，在数据量/用户量达到阈值后启动：

1. 外部搜索引擎、相似题和题目质量治理。
2. 题库开放范围、纠错/举报和题目血缘。
3. 对象存储分片上传、恶意文件扫描和资源版本。
4. 多租户/机构化，仅在 RFC、性能测试、备份和隔离恢复演练通过后实施。

P3，暂不进入近期交付：题库交易市场、购买/结算、机构套餐、家长端、小程序、多端 App 和贡献者体系。

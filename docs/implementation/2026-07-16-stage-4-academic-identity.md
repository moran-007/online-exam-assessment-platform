## 数据模型

- `StudentProfile` 与 `TeacherProfile` 和统一 `User` 一对一，不再维护第二套登录身份。
- `ParentStudent` 只保存显式家长—学生关系，支持主要联系人、解除时间和历史状态。
- `ClassStudent` 增加 `ACTIVE/LEFT`、离班时间、来源和更新时间；移除成员不再物理删除。
- `ClassTeacher` 增加负责人、任课教师、助教角色及同样的生命周期字段。
- `MigrationRun`、`MigrationConflict` 和 `LegacyIdMapping` 记录输入指纹、冲突处置、审批人与旧新 ID 映射。
- `UserStatus.PENDING_ACTIVATION`、`mustChangePassword` 和 `activatedAt` 支持首次激活。迁移器从不查询旧密码字段，为新账号生成不可知随机凭据。

## 权限与数据范围

| 身份 | 档案范围 | 可执行操作 |
| --- | --- | --- |
| 超级管理员 | 全部 | 全部操作，保留平台级兜底 |
| 教务管理员 | 全部 | 维护档案、家长关系、迁移预检/处置/批准 |
| 教师/助教 | 当前任教班级 | 只读学生档案和本人教师档案 |
| 学生 | 本人 | 查看本人档案与已发布 AI 总结 |
| 家长 | 明确关联学生 | 查看关联学生基础档案，不按手机号或姓名推断关系 |

权限拆分为：

- `academic-profile:read`
- `academic-profile:update`
- `parent-student:manage`
- `legacy-migration:manage`

控制器权限、角色限制和服务端数据范围同时生效，前端菜单仅用于展示，不作为安全边界。

## 迁移规则

- 迁移输入使用稳定规范化指纹；同一快照重复预检复用同一演练。
- 只按已存在的 `LegacyIdMapping` 复用对象，姓名相同永远不会触发自动合并。
- 账号、手机号或班级编码冲突先进入 `MigrationConflict`，未处置时禁止批准。
- 真实导入只接受与预检完全相同的输入指纹。
- `LegacyIdMapping` 唯一约束和成员关系 upsert 共同保证重跑不重复建档。
- 旧密码查询数固定为 0；迁入账号均为待激活，管理员重置后变为启用并要求首次改密。

## 第一轮真实演练

- 来源：`worker_01/app/data/attendance.db`
- 演练：`8b52ed30-3984-4395-9a89-8d6aa483459d`
- 输入：46 名学生、16 名教师、32 个可识别家长、17 个班级、51 条班级学生关系、30 个旧账号关联。
- 冲突：5 组重复手机号、2 组同手机号不同家长姓名。
- 处置：重复手机号保留独立身份但不迁移该手机号；家长姓名歧义跳过该家长关系，禁止按姓名合并。
- 结果：`COMPLETED`，177 条映射，92 个待激活账号，51 条班级学生关系，17 条班级教师关系。
- 重跑结果：映射数保持 177，未新增重复用户或关系。

177 条映射由 46 个学生、16 个教师、30 个家长、17 个班级、51 条学生花名册关系和 17 条班级教师关系组成。

## 页面与点击验收

- 管理员编辑学生学号、学校、年级并保存。
- 管理员编辑教师工号和任教学科并保存。
- 管理员显式关联家长与学生；家长登录后只能看到该学生。
- 迁移页面查看阻断原因、填写处置依据、签字并批准 Gate C。
- 班级花名册添加负责人，学生离班后原关系转为 `LEFT`，再次加入时恢复同一关系。
- 学生个人页可查看更新后的教务档案；教师和家长范围均经过后端验证。

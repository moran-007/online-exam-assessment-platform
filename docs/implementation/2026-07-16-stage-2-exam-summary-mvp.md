# 阶段 2：考试总结 MVP 实施记录

> 日期：2026-07-16  
> 状态：完成；DeepSeek 真实模型验收与教师 Gate A 评估已通过

## 已完成

- `ExamSummaryDatasetBuilder` 复用统计查询，先做考试范围鉴权，再生成不含个人身份和答题正文的事实快照。
- 参与、成绩、分布、题目和知识点数值均绑定 `evidenceRef`；输入哈希排除采集时间，保证同一事实集稳定命中。
- `GET /api/v1/ai-summaries/exams/:examId/preview` 在模型不可用时仍可展示确定性统计。
- `POST /api/v1/ai-summaries/exams` 支持自动选择默认配置，也支持手动选择当前用户可使用的模型配置。
- 考试总结 Prompt v1 和 JSON Schema 作为受审版本写入数据库；只有记录审核人后模板才能启用。
- 任务持久化输入快照、可空的请求 Token 上限、预算预留上界、模型/模板/Schema 版本和 correlationId。
- 唯一幂等键覆盖事实哈希、数据集、Prompt、Schema、Provider 和模型；成功结果复用不再次调用或计费。
- 模型调用不放在数据库事务内；结构化输出和证据引用全部通过后，才在短事务中原子写入总结草稿与成功状态。
- 超时、空正文或连接中断若可能已产生费用，写入“用量未报告”流水；本地预算按解析后的预留上界保守预留：优先使用显式请求限制，否则使用模型能力上界，能力未知时使用 8,192 平台校验上界，不把未知费用当作零消耗。
- 任何生成结果都只创建 `draft`，任务执行链没有自动发布路径。
- 教师可编辑草稿、人工审核、发布、撤回和显式重新生成；版本号采用乐观并发控制，已发布内容必须先撤回才能编辑。
- 显式重新生成使用独立 `generationKey`，既保留来源总结血缘，也不会误命中初次生成的幂等结果。
- 任务、详情、考试历史和学生已发布总结查询接口均完成；学生只能看到自己参加过的考试且已发布的总结。
- 编辑、审核、发布、撤回和重新生成均写入审计日志，数据库约束同时阻止缺失审核人或发布时间的非法状态。
- 统计页提供 AI 总结工作台：支持自动选择默认模型、手动选择可用配置、版本历史、证据索引、Token 用量和本地剩余额度展示。
- 学生考试结果页仅展示已发布总结；AI 查询失败不会影响确定性考试成绩与答题回顾。
- 50 组脱敏黄金样本以 JSONL 保存，并逐例校验数据集、输出 Schema、证据引用和脱敏规则。

## 真实模型受控验收

- 阿里配置连接成功；连接测试和两次通用文本总结共返回 1,061 个实际 Token。
- 为阿里测试配置设置 10,000 Token 本地月度预算，调用考试总结前可计算剩余为 8,939 Token。该余额是系统本地预算，不代表供应商账户余额。
- 对含 2 份已评分答卷、5 道题、5 个知识点和 54 条 EvidenceRef 的测试考试发起一次 1,000 输出 Token 上限调用；供应商在配置的 30 秒超时内未返回，用量未知，未创建草稿。
- 阿里考试总结在 30 秒和 60 秒窗口内均出现超时；Key 已通过连接测试和两次通用总结，因此不是无效密钥，但当前 `qwen3.7-plus` 不作为考试总结可用模型继续重试。阿里配置现显示实际已用 1,061、保守预留 4,200、可计算剩余 4,739 Token。
- 一份 OpenRouter 路由 Key 被误配到腾讯云官方混元端点，连接测试返回 HTTP 401、记录用量为 0；该配置判定为端点错误，不重试。系统已提供独立的 OpenRouter / 腾讯混元 Hy3 预设。
- 智谱 GLM 与火山方舟尚未在本地配置页保存，因此没有发起调用，也没有 Token 消耗。
- DeepSeek 连接测试实际使用 17 Token。Prompt v1 阶段的两次考试总结返回了实际用量但未形成草稿：一次达到 1,000 输出 Token 上限，另一次返回 792 输出 Token 但缺少 `schemaVersion`；另有两次旧调用未报告用量。
- DeepSeek V4 默认开启思考模式，历史 1,000 Token 显式上限曾可能被思考过程占用；考试总结、通用总结和连接测试随后显式关闭 DeepSeek 思考模式。后续修正也对 Qwen/阿里云结构化非流式请求发送 `enable_thinking: false`；此前阿里超时记录保持不变，但不再代表修正后的请求策略。网关先提取用量再检查空正文，避免丢失供应商已经返回的 Token 统计。
- Prompt v2 提供完整紧凑 JSON 骨架、条目数量约束、缺失 `schemaVersion` 的安全补全，并为 DeepSeek V4 登记 JSON Object 能力。后续统一修正为配置级与单次生成上限都可留空；两者都为空时不发送 `max_tokens`，存在显式值时取调用级、配置级与模型能力上限的最小值。8,192/能力上界只用于显式输入校验和 usage 未报告时的保守预算预留，1,000 Token 仅作成本参考。失败后再次调用必须二次确认。
- DeepSeek 使用 Prompt v2 成功生成真实考试草稿：请求上限 1,200，输入 6,120、输出 1,061、合计 7,181 Token；生成 8 条结论、25 个 EvidenceRef，未知引用为 0。草稿已完成一次人工编辑、审核并发布，证明完整生命周期可用。
- DeepSeek 本地月度预算为 1,000,000 Token；当前显示实际已用 20,602、保守预留 2,000、可计算剩余 977,398 Token。修复后的预算策略允许在扣除预留后的余额内继续使用，不再因一次未知用量永久锁定整月。
- 失败任务会在总结对话框中持续显示脱敏错误，并明确区分“实际输入/输出”“用量未报告/已预留”和“未形成用量记录”。

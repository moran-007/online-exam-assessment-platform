# 教务系统部署与运维参考

## 定位与边界

教务能力已并入融合平台，使用统一账号、课程、班级、权限、文件、通知和审计。当前可靠部署单元仍是同一个 NestJS 后端、Vue 前端和 PostgreSQL 数据库；不要把教务数据复制到第二套可写数据库，也不要恢复已归档 worker 写入口。

教务侧主要页面：

- `/courses`、`/classes`、`/academic-profiles`
- `/teaching-operations`、`/learning-portal`
- 融合看板中的出勤、课时、教学记录和班级指标

教务侧主要 API 领域：课程与班级、教务档案、课型与课程单元、排课、考勤、课时台账、教学记录、Scratch 课堂和学习门户。课程主键同时被考试、题目、试卷和班级引用，是教务与测评的共享契约。

## 部署模式

### 统一部署（推荐）

使用仓库的一键部署或本地一键启动。端口保持不变：后端 3000、前端开发入口 5173，生产环境由 Nginx 对外提供 80/443。

本地启动：

```powershell
scripts\start-platform.ps1 -NoBrowser
```

生产部署见 `deploy/ONE_CLICK_DEPLOY.md`。数据库迁移、附件目录、备份和密钥由统一平台管理。

### 教务入口按需开放

需要给教务团队单独入口时，仍部署同一构建产物和同一 API，可使用独立域名或路径指向同一 Nginx upstream，并给账号只分配教务权限。导航会按权限收敛；超级管理员始终拥有全平台能力，不适合作为受限教务账号。

该模式只分离入口与职责，不改变端口、路由事件、数据库所有权或写模型。它可避免课程、班级和考试关联发生双写分歧。

### 物理拆分限制

当前版本不支持把教务模块作为独立数据库服务单独写入。若未来必须物理拆分，应先定义统一身份、课程、班级、文件和审计契约，再引入事件出站箱与幂等消费；在完成一致性、回放、备份和灾难恢复设计前，不得通过复制表或双写实现。

## 最小权限集合

按岗位组合权限，不要直接授予全量权限：

- 教务只读：课程、班级、档案、排课、考勤、课时、教学记录读取。
- 排课人员：增加 `schedule:manage`。
- 任课教师：增加考勤确认/更正、教学记录管理和所需 Scratch 权限。
- 财务或校区运营：增加课时调整、对账；不可直接修改历史流水。
- 学生与家长：仅本人或关联学生范围的课表、课时、已发布教学内容。

后端守卫和数据范围是最终边界；隐藏页面按钮不能替代权限校验。

## 发布前验收

1. 创建并编辑课程，确认班级关联同一课程主键。
2. 创建一级和子知识点，完成查看、修改、先子后父删除。
3. 生成课次，执行调课、取消或补课，确认原记录和审计仍存在。
4. 上传素材、替换素材、修改教学记录并发布，学生端只能看到公开字段和公开附件。
5. 考勤确认只扣一次；更正产生冲正流水；全量重算差异为零。
6. 学生和家长数据范围拒绝越权，教师只能访问任教班级。
7. 检查 `/api/v1/health`、日志、通知、附件哈希、备份与隔离恢复报告。

浏览器闭环由 `test/e2e/core-browser-flows.spec.ts` 的 Gate D 和课程子项用例覆盖；后端调课、取消、补课由教务集成测试覆盖。

## 备份、回滚与故障处理

教务数据与测评数据共享事务和外键，必须整库备份并连同 uploads 一起验证。回滚使用已验证备份恢复到隔离库后再切换，不能只回滚教务表。课时余额由不可变台账汇总，纠错只能追加调整或冲正。

常见问题、日志路径、Token、导出和附件排查见 `troubleshooting-and-maintenance.md`；worker 切换状态见 `worker-cutover-runbook.md`。

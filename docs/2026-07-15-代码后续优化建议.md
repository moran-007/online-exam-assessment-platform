# 代码后续优化建议

> 审计日期：2026-07-15
> 审计分支：`codex/p1-architecture-refactor`
> 审计提交：`f2bb366315a9aa61a79849abbd1f75023b6a074b`
> 适用项目：在线考试与教学管理平台

## 1. 执行结论

项目当前技术路线是合理的：继续使用 NestJS + Prisma + PostgreSQL 构建模块化单体，前端继续使用 Vue 3 + Vite，不建议在现阶段引入微服务、Nx/Turborepo、全局状态库或重建数据库。

刚完成的 P1 重构已经解决了四个最紧迫的问题：题库、学生考试、Hydro、导出巨型 Service 已按用例拆分；重点 Vue 页面具备 feature 目录和页面编排层；路由与重型依赖完成懒加载/分包；Swagger、Orval 和类型化请求适配器已经落地。因此下一阶段不应再次做大范围目录搬迁，而应优先补齐“门禁、安全、类型和可靠性”。

建议按以下顺序推进：

1. **P0：让现有架构门禁真正进入 CI，并解决 Hydro 明文凭据。**
2. **P0：消除 P1 页面中的 `@ts-nocheck`，让类型化 API 真正约束页面状态。**
3. **P1：拆分仍然超大的试卷、用户、阅卷 Service 和页面 composables。**
4. **P1：建立可量化的测试覆盖、数据库性能和可观测性基线。**
5. **P1/P2：为导出队列、文件存储增加可替换接口；达到多实例需求时再接入 Redis/BullMQ 和 S3/MinIO。**
6. **P2：在教务能力并入时扩展领域模型，不把教务逻辑继续塞进现有 classes/users Service。**

## 4. P0：必须优先完成

### 4.1 将完整质量门禁接入 GitHub Actions

#### 现状

根 `package.json` 已提供：

- `architecture:check`
- `openapi:check`
- `frontend:typecheck`
- `bundle:check`
- 汇总脚本 `test:ci`

但 `.github/workflows/ci.yml` 当前只运行 lint、构建、Jest 和 Playwright，没有运行架构检查、OpenAPI/生成客户端差异检查、`vue-tsc` 和包体积预算。也就是说，本地已具备的 P1 防回退能力没有成为合并前的强制条件。

#### 具体改造

1. 在 CI 中显式增加四个步骤，或在安装浏览器后统一调用经过拆分优化的 `test:ci`。
2. OpenAPI 生成后执行 `git diff --exit-code -- docs/openapi.json frontend/src/api/generated`。
3. `frontend:typecheck` 独立展示，便于定位类型回归。
4. `bundle:check` 上传 `dist/.vite/manifest.json` 和构建统计作为失败证据。
5. 保持 Prisma 空库迁移和 drift 检查；增加 seed smoke test 时必须使用测试数据，不复用生产导出。
6. 将 main 分支保护设置为必须通过该工作流后才能合并。

#### 验收标准

- 人为修改 DTO 而不重新生成客户端时，CI 必须失败。
- 将路由恢复为静态导入、引入旧 `api()` 或生成超预算 chunk 时，CI 必须失败。
- TypeScript 页面字段访问错误必须在 `vue-tsc` 阶段失败。
- CI 产物和本地 `pnpm test:ci` 结果一致。

### 4.2 加密 Hydro 外部账号凭据

#### 现状与风险

`HydroAccount.loginPassword` 当前以可逆业务需要为由直接存储文本，并在登录时读取。第三方账号密码与本平台密码风险不同：无法只保存哈希，但也绝不能明文落库。数据库备份、开发快照或 SQL 只读权限泄露，都可能导致用户 Hydro 账号被直接接管。

#### 目标设计

采用应用层信封加密：

- 算法：AES-256-GCM；每条记录使用独立随机 96-bit IV。
- 存储：`ciphertext`、`iv`、`authTag`、`keyVersion`，不再存 `loginPassword` 明文。
- 主密钥：仅从部署环境的 Secret 注入；不写入 `.env.example` 的真实值，不进入数据库和日志。
- 访问：只允许 Hydro 登录用例通过 `CredentialCipher` 解密；Controller、DTO、日志和响应均不得拿到明文。
- 轮换：通过 `keyVersion` 支持旧密钥解密、新密钥写入；提供一次性后台迁移脚本。

建议新增：

- `src/security/credential-cipher.service.ts`
- `src/security/credential-cipher.module.ts`
- Prisma 字段迁移和幂等迁移脚本
- 日志脱敏规则：password、token、cookie、authorization、credential、secret

#### 迁移顺序

1. 新增密文字段但暂不删除旧字段。
2. 部署支持“优先读密文、仅迁移阶段回退明文”的版本。
3. 离线批量加密并核对成功数、失败数和不可解密账号。
4. 禁止新明文写入，观察一个发布周期。
5. 删除明文字段和回退路径；执行备份保留策略，避免旧明文备份无限期存在。

这里允许短期双字段只作为受控数据迁移，不允许形成长期双轨业务实现。

#### 验收标准

- 数据库、API、Pino 日志和错误栈中均不可检索到测试账号密码。
- 错误密钥、篡改 auth tag、未知 key version 均明确失败且不触发第三方登录。
- 密钥轮换前后的账号登录、机器人阻断和判题回写测试全部通过。

### 4.3 消除业务 `@ts-nocheck`

#### 现状

以下 composables 仍禁用了 TypeScript 检查：

- questions：`useQuestionPage.ts`、`useQuestionImportPage.ts`
- exams：`useExamTakingPage.ts`、`useExamManagementPage.ts`、`useStatisticsPage.ts`
- hydro：`useExternalAccountPage.ts`
- exports：`useExportPage.ts`

`components.d.ts` 的生成文件可按工具约定处理；业务文件必须逐个清零。

#### 具体改造

1. 先为每个 feature 建立稳定的领域模型，不直接把 Orval 生成类型散布到模板：
   - `QuestionListItem`、`QuestionDraft`、`QuestionImportRow`、`DuplicateCandidate`
   - `ExamSessionState`、`AnswerDraft`、`AutosaveStatus`、`RemoteJudgeState`
   - `ExportTaskViewModel`、`HydroAccountStatus`
2. 在 feature API 层把生成 DTO 映射为页面模型，集中处理 nullable、时间、枚举和后端兼容字段。
3. 将 `Record<string, any>` 替换为明确接口、判别联合或 `unknown` + 类型守卫。
4. 表单引用使用 Element Plus 的公开类型；定时器统一用 `ReturnType<typeof setTimeout>`。
5. 每移除一个 `@ts-nocheck`，加入对应 composable 单元测试，再进入下一文件。
6. 扩展架构检查：业务目录出现 `@ts-nocheck`、显式 `any` 超预算或视图直接导入 generated API 时失败。

#### 验收标准

- 7 个业务 `@ts-nocheck` 全部删除。
- 手工业务代码不新增无说明的 `any`。
- `pnpm frontend:typecheck` 纳入 CI 且稳定通过。
- API 字段被删除或变为 nullable 时，在编译或映射层测试中被发现，而不是运行到页面才报错。

## 5. P1：剩余模块拆分

### 5.1 试卷领域

`papers.service.ts` 是当前最大后端文件，应成为下一轮拆分的第一目标。

建议拆为：

```text
src/modules/papers/
  queries/
    list-papers.use-case.ts
    get-paper-detail.use-case.ts
    preview-paper.use-case.ts
  commands/
    create-paper.use-case.ts
    update-paper.use-case.ts
    delete-paper.use-case.ts
    publish-paper.use-case.ts
    copy-paper.use-case.ts
  assembly/
    paper-question-assembler.ts
    paper-score-validator.ts
  snapshots/
    paper-snapshot.builder.ts
  policies/
    paper-access.policy.ts
```

约束：

- Controller 直接依赖用例，不增加新的 `PapersFacade`。
- 试卷发布、题目快照和总分校验由同一事务用例持有。
- 学生预览只依赖窄的 `PaperPreviewReader`，不能导入整个写 Service。
- 题库快照通过已存在的窄接口调用，不跨模块访问题库内部 Prisma 查询。

### 5.2 用户与权限领域

教务系统会新增学生档案、教师档案和家长关系。不能把这些内容继续堆入 `users.service.ts`。

建议保留 `users` 只负责：

- 账号生命周期、状态和基础身份信息；
- 密码重置与登录安全状态；
- 用户列表的通用查询。

角色分配、权限目录、数据范围分别由现有授权模块的用例持有；教务档案进入独立 `student-profiles`、`teacher-profiles`、`parent-relations` 模块。用户创建与档案创建需要一个明确的应用层事务用例，但不把所有档案字段复制到 User。

### 5.3 阅卷领域

`grading.service.ts` 建议拆为：任务分配、待阅查询、主观题评分、复核/驳回、批量提交、评分汇总、评分可见性。评分写入必须保留乐观并发控制或版本检查，避免两名阅卷人覆盖。

验收应重点覆盖：

- 同一答题项重复提交评分的幂等性；
- 分数上下界、总分重算和考试结算事务；
- 管理员、阅卷教师、非授权教师的数据范围；
- 复核后排名和统计缓存失效。

### 5.4 前端二次拆分

P1 第一轮把路由 View 逻辑移动到 composable，降低了 SFC 体积，但两个题库 composable 仍是“巨型页面 Service”。下一步按状态生命周期拆，而不是按函数数量机械分文件。

`useQuestionImportPage.ts` 建议拆为：

- `useSingleQuestionDraft`
- `useBatchQuestionImport`
- `usePortableImport`
- `useQuestionAssets`
- `useDuplicateDetection`
- `useMaterialChildren`
- `useHydroQuestionReference`

页面级 composable 只组合这些能力和协调跨区域提交。

`useQuestionPage.ts` 建议拆为：

- `useQuestionFilters`
- `useQuestionSelection`
- `useQuestionEditor`
- `useQuestionPractice`
- `useQuestionExport`
- `useQuestionHydroBinding`

`PaperView.vue` 需要迁入 `features/papers`，拆出列表筛选、试卷编辑器、题目编排、分值校验、预览和发布状态组件。路由 View 目标继续保持在约 500 行以内，核心 composable 建议控制在 300–500 行。

## 6. P1：测试策略升级

### 6.1 当前缺口

项目现有 5 个单元测试文件、1 个集成测试文件、1 个 Playwright 文件。它们能提供关键冒烟保护，但与 197 个 API 操作、49 个数据模型和多个角色流程相比仍不足；Jest 也没有覆盖率阈值。

### 6.2 建议测试金字塔

| 层级 | 覆盖对象 | 要求 |
| --- | --- | --- |
| 纯单元测试 | 解析器、渲染器、评分规则、权限策略、DTO 映射、时间计算 | 无数据库，快速且确定 |
| 用例集成测试 | 事务、幂等、状态机、权限和副作用 | 使用独立 PostgreSQL 测试库 |
| 契约测试 | OpenAPI 与真实响应、下载头、错误结构 | 对每个 tag 至少覆盖代表操作 |
| 浏览器测试 | 管理员、教师、学生、公开用户关键旅程 | 只测高价值跨层流程，不复制所有 API 用例 |

优先补齐：

1. 试卷发布/快照/跨题库引用。
2. 阅卷并发、复核和排名重算。
3. Hydro 凭据迁移、机器人阻断、轮询恢复。
4. 导出任务在进程重启后的状态收敛。
5. 教务并入后的排课、考勤、课消和家长数据范围。

覆盖率不应一步设成不现实的 100%。建议先对新增/改动代码设定 80% 行覆盖和 70% 分支覆盖，核心策略/状态机要求 90% 分支覆盖；随后再提升全仓基线。

## 7. P1：数据库与查询优化

### 7.1 建立基线，而不是先盲目加缓存

先记录以下指标：

- P50/P95/P99 API 延迟和数据库时间；
- 每个请求 SQL 数量；
- 慢查询 Top 20；
- 题目列表、考试结果、统计报表、导出数据装载的扫描行数；
- PostgreSQL 连接池等待和长事务。

对 Top 查询使用 `EXPLAIN (ANALYZE, BUFFERS)`，再决定复合索引。不要仅根据单字段 where 条件堆叠索引；索引顺序要匹配实际过滤、排序和数据选择性。

### 7.2 具体建议

- 列表查询统一游标或稳定分页策略，排序必须包含唯一键，避免翻页重复。
- 统计聚合不要在 Controller 或前端拼装；固定口径放入查询用例。
- 大导出使用分批/流式读取，避免把全量题库、答卷或附件一次性装入内存。
- 对考试提交、考勤课消等关键写入缩短事务范围，不在数据库事务中等待外部 HTTP。
- 为数据保留和清理建立任务：过期刷新令牌、导出文件、孤立资源、旧判题轮询记录。
- 教务模型引入后，为 `LessonSession(classId, startsAt)`、`AttendanceRecord(sessionId, studentId)`、`LessonHourLedger(studentId, occurredAt)` 等按真实查询建立复合索引和唯一约束。

## 8. P1：可观测性与运维

### 8.1 指标

建议先接入 Prometheus 兼容指标，不必同时引入完整可观测性平台：

- HTTP 请求数、错误率和延迟；
- Prisma 查询耗时和连接池状态；
- Hydro 登录/阻断/提交/轮询成功率；
- 导出排队长度、处理耗时、失败/重试数；
- 考试自动保存失败、提交冲突、强制结束数；
- 后续考勤写入、课消失败和数据校验异常数。

### 8.2 追踪和日志

- 为请求、后台任务、Hydro 提交、导出任务使用统一 `correlationId`。
- 日志只记录业务标识、状态和耗时，不记录答案正文、密码、Token、Cookie 和完整第三方响应。
- 审计日志与运行日志分开：审计日志记录“谁在何时改变了什么”，运行日志用于诊断。
- 关键写用例在同一事务内写业务数据与 outbox 事件；需要可靠异步处理时再启动 outbox worker。

### 8.3 告警

先建立少量可行动告警：

- 5 分钟错误率超过阈值；
- 导出任务长时间处于 processing；
- Hydro 机器人阻断率突增；
- 自动保存连续失败；
- 数据库连接池耗尽或迁移失败；
- 附件磁盘剩余空间低。

## 9. P1/P2：任务队列和文件存储的演进边界

### 9.1 导出队列

当前导出任务由应用进程内轮询/恢复，适用于单实例和可接受短暂恢复的环境。立即全面引入 Redis 会增加部署和故障面，因此建议分两步：

1. 现在抽象 `ExportJobQueue`，实现仍使用 PostgreSQL 任务表 + 单实例 worker，并增加租约、心跳、幂等和启动恢复。
2. 当出现多实例、任务必须跨重启可靠执行、吞吐瓶颈或定时任务竞争时，将实现替换为 BullMQ + Redis。

无论使用哪种队列，任务处理器都必须可幂等：相同 task id 重放不能生成重复状态、副作用或无限附件。

### 9.2 文件存储

当前本地文件系统适合单机部署。先抽象：

```ts
interface ObjectStorage {
  put(input: PutObjectInput): Promise<StoredObject>;
  open(key: string): Promise<Readable>;
  delete(key: string): Promise<void>;
  createDownloadUrl(key: string, expiresInSeconds: number): Promise<string>;
}
```

同时记录 SHA-256、size、mimeType、owner、purpose 和生命周期状态。当应用多实例、共享附件、异地备份或 CDN 成为需求时，再将实现切换到 MinIO/S3。业务模型不得保存本机绝对路径。

## 10. OpenAPI 与前端契约继续收紧

生成客户端已经解决了 URL 和请求字段漂移，但仍需避免“所有返回值都退化为通用对象”。建议：

- 为分页、详情、状态转换、批量结果、文件元数据定义明确响应 DTO。
- `ApiEnvelope<T>` 只负责协议外壳；`data` 内部必须是领域 DTO，不使用 `Record<string, unknown>` 掩盖字段。
- 错误结构固定包含 `code`、`message`、`details?`、`requestId`，HTTP 状态与业务错误码分工明确。
- 文件下载在 OpenAPI 中声明 content type 和 binary schema；前端只通过类型化 Blob 适配器调用。
- 按 tag 检查 Controller 的 `@ApiResponse`/`@ApiOkResponse`，逐步减少生成客户端中的匿名/宽泛类型。
- 前端时间、Decimal、JSON 字段通过映射层转换，避免组件直接解释后端序列化细节。

## 11. 安全补强清单

除 Hydro 凭据外，还建议：

- Access Token 短时有效，Refresh Token 轮换并能按会话撤销；检测旧 Refresh Token 重用。
- 登录、验证码、密码重置、公开题库和文件下载实行分层限流。
- 上传文件以内容检测、扩展名、MIME、大小和权限共同校验；下载使用短时签名或鉴权流。
- 生产 CORS、Swagger、日志级别由环境白名单控制，启动时拒绝弱默认 Secret。
- Prisma 原始查询集中审计，禁止用户输入拼接 SQL。
- 定期执行依赖漏洞扫描和许可证清单；高风险升级通过回归测试后合并。
- 对角色、权限、班级范围、学生结果可见性建立拒绝路径测试，而不只测成功路径。

## 12. 与教务系统并入的衔接原则

教务能力会带来排课、考勤、课时台账、学生/教师档案、家长关系和 Scratch 课堂作品。建议遵守：

- 复用现有 User、Role、Permission、Course、ClassGroup、FileAsset、AuditLog、Notification。
- 新领域使用独立模块和 Prisma model，不扩充成新的 `users.service.ts`/`classes.service.ts` 巨型文件。
- 课消使用不可变台账，不直接在学生记录上反复减余额。
- Scratch/Hydro 专用运行时通过适配器或事件连接，不与 NestJS 进程强行合并。
- 迁移只允许短期受控兼容层；切换完成后删除 Flask 业务实现，不维护永久双写。

详细模型、数据映射和迁移方案见《2026-07-15-教务系统融入可行性与技术方案.md》。

## 13. 建议执行路线图

### 迭代 1：守住现有成果

- CI 接入 architecture、OpenAPI、vue-tsc、bundle 全部门禁。
- Hydro 凭据加密设计、Schema 迁移和回归测试。
- 建立当前性能、错误率、包体积和测试覆盖基线。

### 迭代 2–3：类型和核心服务

- 清理题库两个巨型 composable 的 `@ts-nocheck` 并二次拆分。
- 拆分 `papers.service.ts`，冻结试卷发布/快照行为。

### 迭代 4–5：用户、阅卷与运维

- 拆分 `users.service.ts` 和 `grading.service.ts`。

## 14. 验收指标

| 维度 | 目标 |
| --- | --- |
| 架构 | 不存在新增 >500 行用例/服务；路由 View 以编排为主；无旧业务 Facade 回流 |
| 类型 | 业务 `@ts-nocheck` 为 0；业务 `any` 有预算且持续下降；`vue-tsc` 必过 |
| 契约 | OpenAPI/客户端生成无差异；页面不拼业务 URL；响应 DTO 不退化为通用对象 |
| 测试 | 核心事务、角色拒绝路径、外部副作用均有测试；改动代码达到覆盖阈值 |
| 性能 | 核心 API 有 P95 基线；无确认的 N+1；大导出内存稳定 |
| 安全 | Hydro 密码不明文；敏感信息不进入日志/响应；弱 Secret 启动失败 |
| 可靠性 | 导出任务可幂等恢复；文件有校验和与生命周期；迁移可回滚 |
| 运维 | 请求/任务可通过 correlation id 追踪；关键失败有可行动告警 |

## 15. 明确不建议做的事

- 不因模块增多立即拆微服务；当前事务边界和团队规模更适合模块化单体。
- 不引入第二套认证、第二套用户表或教务专用登录系统。
- 不把 Flask/SQLite 代码直接复制进 NestJS，也不长期保留两套教务写入口。
- 不在没有性能证据时大量加 Redis 缓存。
- 不把 Orval 生成类型直接当作所有页面状态模型。
- 不删除历史 Prisma migration、生产数据兼容或既有对外 API。
- 不以“文件已拆小”为完成标准；用例边界、事务所有权、依赖方向和测试才是核心标准。

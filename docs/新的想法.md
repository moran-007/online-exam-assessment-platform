
1. **题量不限**：百万级题库也能导入、检索、组卷、统计。
2. **题型不限**：以后可以加任意题型，不需要大改数据库和判分逻辑。
3. **内容不限**：文字、图片、音频、视频、代码、Scratch、Arduino、文件、材料题、交互题都能承载。
4. **场景不限**：考试、练习、作业、闯关、竞赛、测评、错题复习、自适应推荐都能复用同一套底层能力。

下面是我认为**目前还需要补齐的关键功能**。

---

## 一、最核心：题型引擎化，而不是继续硬编码题型

PRD 里已经要求题型可扩展、判分规则可扩展、Hydro 可替换、AI 服务可替换。 但要真正做到“不限题目”，你需要把题型从业务代码里抽象出来。

建议新增一个 **题型注册中心**：

```txt
question_type_registry
```

每种题型都定义：

| 能力          | 说明                     |
| ----------- | ---------------------- |
| 数据结构 schema | 这类题目需要哪些字段             |
| 编辑器组件       | 老师后台怎么录入               |
| 学生答题组件      | 学生端怎么展示和作答             |
| 答案结构        | 学生答案如何保存               |
| 判分策略        | 自动判分、人工判分、AI 辅助判分、外部判分 |
| 统计适配器       | 正确率、得分率、选项分布、知识点统计怎么计算 |
| 导出适配器       | PDF、Word、Excel 怎么导出    |
| 预览组件        | 后台预览、学生模拟答题怎么渲染        |

否则你后面每加一种题型，比如材料题、连线题、排序题、拖拽题、Scratch 项目题、Arduino 实物题，都要改很多地方。

**建议优先补的题型：**

1. 材料题 / 阅读理解题，支持一个材料下面挂多个子题。
2. 多空填空题，支持每空独立判分。
3. 排序题。
4. 匹配题 / 连线题。
5. 文件上传题。
6. 音视频作答题。
7. 编程题的多 provider 抽象。
8. Scratch / Arduino 项目题。
9. 问答题 rubrics 评分题。
10. 组合题：一道大题里包含选择、填空、简答、编程等子题。

---

## 二、补“通用题目数据模型”

现在 PRD 里的题目字段已经比较完整，包括题干、题型、选项、答案、解析、分值、难度、课程、知识点、标签、状态、版本号、是否允许选项随机等。 但如果要“不限题目”，建议把题目模型进一步拆成三层：

```txt
Question
  ├─ 基础元数据：课程、知识点、标签、难度、状态、创建人
  ├─ 内容结构：题干、材料、附件、子题、渲染配置
  └─ 判分结构：标准答案、评分规则、评分插件、人工评分 rubric
```

也就是说，不要让所有题型都挤在固定字段里。建议核心表只保存通用字段，题型差异放到 JSON schema 或扩展表里。

推荐结构：

```txt
questions
question_versions
question_type_definitions
question_assets
question_options
question_answers
question_scoring_rules
question_sub_items
```

其中 `question_versions` 非常重要。你已经有“已发布考试使用题目快照”的方向，后续要加强成完整快照机制：**考试发布时必须冻结题目内容、选项顺序、答案规则、分值、解析、资源引用版本**。

---

## 三、补“材料题 / 组合题”能力

如果目标是不限题目，**材料题是必须补的**。因为很多复杂题型都可以通过“材料 + 子题”表达。

例如：

```txt
材料：阅读一段 Python 代码
  子题 1：单选题
  子题 2：填空题
  子题 3：简答题
  子题 4：编程改错题
```

这会影响很多模块：

| 模块   | 需要支持             |
| ---- | ---------------- |
| 题库   | 大题、子题、材料、附件      |
| 组卷   | 按大题抽取，还是按子题抽取    |
| 试卷展示 | 大题连续展示，子题独立答题    |
| 判分   | 子题独立计分，大题汇总      |
| 统计   | 子题正确率、大题得分率      |
| 错题本  | 收录子题，关联原材料       |
| 导出   | Word/PDF 中保持大题结构 |

这个能力比单独加几个小题型更关键。

---

## 四、补“判分引擎”

目前客观题自动判分和主观题批改已经有基础，PRD 里也提到多选题评分模式、填空题评分规则、主观题人工批改等。 但要无限扩展题型，需要把判分抽象成独立引擎。

建议设计：

```txt
ScoringEngine
  ├─ choice_strict_scorer
  ├─ multi_choice_partial_scorer
  ├─ blank_text_scorer
  ├─ blank_numeric_scorer
  ├─ regex_scorer
  ├─ rubric_manual_scorer
  ├─ code_judge_scorer
  ├─ file_manual_scorer
  └─ ai_assisted_scorer
```

每次判分都保存：

```txt
score
max_score
scoring_status
scoring_method
scoring_rule_snapshot
scoring_version
scored_by
scored_at
score_detail
```

特别要补三个能力：

1. **重判能力**：题目答案或评分规则修正后，可以对指定考试、指定题目、指定学生批量重判。
2. **评分规则版本化**：否则历史成绩不可追溯。
3. **评分详情**：不只是给分，还要知道为什么给这个分。

---

## 五、补“AI 学情分析最小闭环”

PRD 当前状态里写得很清楚：AI 数据库结构和报告模型已预留，但暂未接入真实 AI 服务和报告生成策略；下一阶段优先需求也是 AI 学情分析最小闭环。

建议先不要做很复杂的 AI。先做一个能落地的闭环：

```txt
考试结果 → 知识点得分 → 错题归因 → 生成学生报告 → 教师审核 → 发布给学生
```

第一版 AI 功能建议包括：

| 功能       | 价值                   |
| -------- | -------------------- |
| 学生单次考试报告 | 给学生看本次表现             |
| 知识点薄弱分析  | 找出真正不会的点             |
| 错题归因     | 粗心、概念不清、代码理解错误、语法问题等 |
| 推荐练习     | 从题库里按知识点和难度推荐        |
| 教师讲评建议   | 帮老师快速备讲评课            |
| AI 输出审核  | 老师可编辑后发布             |

这里要注意：**AI 不应该直接改成绩**。AI 可以做解释、建议、归因、推荐，但正式分数仍应来自判分系统或人工批改。

---

## 六、补“推荐练习 / 自适应练习”

你现在已经有错题本和复习提醒，但如果目标是“不限题目”，下一步应该做推荐系统，而不只是错题复练。

推荐逻辑可以分三层：

```txt
第一层：错题推荐
第二层：同知识点相似题推荐
第三层：按掌握度自适应推荐
```

建议新增学生知识点掌握度表：

```txt
student_knowledge_mastery
```

字段包括：

| 字段                     | 说明        |
| ---------------------- | --------- |
| student_id             | 学生        |
| course_id              | 课程        |
| knowledge_id           | 知识点       |
| mastery_score          | 掌握度 0-100 |
| attempt_count          | 作答次数      |
| correct_rate           | 正确率       |
| recent_correct_rate    | 近期正确率     |
| last_practiced_at      | 最近练习时间    |
| weakness_reason        | 薄弱原因      |
| recommended_difficulty | 推荐难度      |

这样你后面可以做：

1. 每日推荐练习。
2. 薄弱点专项练习。
3. 班级高频错题练习。
4. 考前冲刺卷。
5. 学生个性化随机卷。
6. 错题变式训练。

---

## 七、补“公开题库 / 题库开放范围”

PRD 当前实现状态里提到，公开题库开放策略还需要增加“全公开、仅登录、仅课程学员”等范围字段。 这对“不限题目”很重要。

建议题目增加可见范围：

```txt
private              仅创建者
school               本机构
course_students      课程学员
login_users          登录用户
public               全公开
marketplace          题库市场
```

同时要加：

1. 题目引用次数。
2. 题目被考试占用状态。
3. 题目复制 / 派生关系。
4. 题目来源。
5. 版权 / 授权字段。
6. 题目质量评分。
7. 举报 / 纠错机制。
8. 审核流。

如果未来题目数量很大，这部分会决定平台能不能从“内部工具”变成“题库平台”。

---

## 八、补“大规模题库检索和去重”

当前题库已经支持分页、排序、标签/知识点筛选、重复检测。 但“不限题目”需要更强的检索能力。

建议补：

1. 全文搜索：题干、解析、代码、标签、知识点。
2. 高级筛选：课程、知识点、题型、难度、来源、状态、创建人、更新时间、使用次数。
3. 相似题检测：导入时、编辑时、发布审核时都检测。
4. 题目质量分：错误率过高、争议题、长期无人答对的题自动提示。
5. 批量治理：批量改标签、批量归档、批量审核、批量迁移知识点。
6. 题目血缘：原题、改编题、变式题、同源题。

这部分建议后续接搜索引擎，例如 Meilisearch、Typesense、OpenSearch 或 Elasticsearch。前期可以先用数据库全文索引，后期再迁移。

---

## 九、补“资源对象存储和资源版本化”

当前资源还是本地目录，PRD 里也写到后续多实例部署应迁移到 MinIO/对象存储。

如果题目不限，资源一定会越来越多：

1. 图片。
2. 音频。
3. 视频。
4. PDF。
5. Word。
6. Scratch 项目文件。
7. Arduino 代码包。
8. 学生上传作业。
9. 编程题测试数据。
10. 题目附件素材。

建议补：

| 能力         | 说明            |
| ---------- | ------------- |
| 对象存储       | MinIO / S3 兼容 |
| 资源版本       | 题目快照引用资源版本    |
| 引用计数       | 防止误删          |
| 下载审计       | 谁下载过资源        |
| 防盗链 / 权限校验 | 学生不能越权访问答案资源  |
| 大文件上传      | 分片上传、断点续传     |
| 病毒扫描       | 文件上传题必须考虑     |
| 跨环境迁移校验    | 本地、测试、生产资源一致性 |

## 十、补“异步任务体系”

PRD 当前导出队列还是单进程内置队列，后续建议迁移 Redis + BullMQ。 这个必须做，因为你后面会有很多耗时任务。

建议统一任务中心：

```txt
jobs
job_logs
job_results
```

任务类型包括：

1. 题库批量导入。
2. 题库导出。
3. 试卷导出。
4. 成绩导出。
5. AI 报告生成。
6. Hydro 同步。
7. 编程题批量重判。
8. 资源清理。
9. 相似题检测。
10. 统计报表预计算。

任务中心要支持：

```txt
排队、执行中、成功、失败、取消、重试、进度、日志、结果下载、权限快照
```

## 十一、补“多 OJ / 多测评 Provider 抽象”

Hydro 最小闭环已经完成，但 PRD 也提到后续要补 provider 适配器、异步队列化同步和更细的任务看板交互。

建议不要把 Hydro 写死。应该抽象成：

```txt
JudgeProvider
  ├─ HydroProvider
  ├─ LocalJudgeProvider
  ├─ CodeRunnerProvider
  ├─ ScratchJudgeProvider
  └─ ArduinoManualProvider
```

统一接口：

```txt
submitCode()
getSubmissionResult()
syncProblem()
syncSubmission()
rejudge()
normalizeResult()
```

这样以后你可以支持：

1. Hydro。
2. 本地代码运行。
3. 其他 OJ。
4. Scratch 文件人工 / 半自动测评。
5. Arduino 项目提交。
6. 数据结构题、SQL 题、网页题等特殊测评。

---

## 十二、补“考试安全和异常处理”

一期可以不做重监考，但如果要平台化，至少要有轻量考试安全能力。

建议补：

1. 切屏记录。
2. 复制粘贴记录。
3. 离开页面记录。
4. 异常提交记录。
5. IP / 设备记录。
6. 多端登录冲突策略。
7. 答案保存冲突处理。
8. 网络断开恢复提示。
9. 超时后后端强制拒绝保存。
10. 教师端异常学生列表。

不建议一开始做摄像头监考，成本高、争议大、准确率也不一定好。先做行为日志和异常提醒更实际。

---

## 十三、补“字段级 / 按钮级权限”

PRD 当前明确说字段级、按钮级权限仍需继续收敛。

建议你把权限拆成三层：

```txt
菜单权限
接口权限
数据权限
字段/按钮权限
```

例子：

| 场景              | 权限控制    |
| --------------- | ------- |
| 助教能看成绩但不能发布成绩   | 按钮权限    |
| 教师只能看自己班级       | 数据权限    |
| 学生不能看答案解析       | 字段权限    |
| 题库管理员能审核但不能管理用户 | 接口权限    |
| 家长只能看孩子数据       | 关系型数据权限 |

这对后续商业化、学校多租户、家长端都很重要。

## 十四、补“多租户 / 机构化”

如果这个平台以后不只是自己用，而是给多个学校、机构、老师用，就必须提前考虑多租户。

```txt
organizations
campuses
departments
classes
```

```txt
tenant_id / organization_id
```

否则后面想从单机构变多机构，会非常痛苦。

需要考虑：

1. 不同机构题库隔离。
2. 公共题库共享。
3. 机构内课程体系。
4. 机构内老师、学生、班级。
5. 机构管理员。
6. 套餐限制。
7. 数据导出隔离。
8. 品牌配置。
9. 域名 / Logo / 主题色配置。

## 十五、补“数据看板和运营指标”

目前统计分析已经有成绩分布、班级对比、知识点趋势、题目诊断图表等。 但后续平台化还需要运营型指标：

这些指标能帮助你判断平台是不是真的在被使用。

# 最关键的产品判断

你现在最不应该继续零散加功能。
下一步应该把平台从：

```txt
固定题型答题系统
```

升级成：

```txt
题型可插拔 + 判分可插拔 + 内容可扩展 + 资源可扩展 + 统计可适配的测评平台
```

一句话概括：**你缺的不是更多页面，而是“题型引擎、判分引擎、任务引擎、资源引擎、推荐引擎”这五个底座。**

我的建议是：**先做题型引擎和材料题，这是通往“不限题目”的第一步。**
